# MCP (Model Context Protocol) 도구 설계 원칙

본 문서는 LLM 에이전트의 작업 효율과 정확도를 극대화하기 위해 품질 관리(Quality-Check) 도구 및 MCP 서버를 구축할 때 반드시 준수해야 할 4가지 핵심 아키텍처 원칙을 정의합니다.

## 1. 실행의 '경로 독립성' 및 '발견 가능성' (Path Independence)
*   **문제**: 에이전트가 `npx ts-node scripts/check.ts` 와 같이 파일 시스템의 물리적 구조나 상대 경로를 추측하게 하면 `MODULE_NOT_FOUND` 등의 에러가 발생하기 쉽습니다.
*   **원칙**: 도구는 반드시 추상화된 MCP 명령(예: `quality-check`)으로 등록되어야 합니다. 에이전트는 도구의 내부 바이너리 위치에 신경 쓰지 않고, 오직 '대상 파일'과 '분석 옵션' 등 본질적인 인터페이스에만 집중할 수 있어야 합니다.

## 2. '상태 동기화(State Sync)'의 가시성 확보 (Visibility of State)
*   **문제**: 캐시가 갱신되지 않거나 커버리지 리포트가 만료되었을 때, 도구가 단순히 "실패"만 반환하면 에이전트는 코드의 결함인지 환경의 결함인지 구분하지 못하고 헛된 수정을 반복하게 됩니다.
*   **원칙**: 도구는 리포트의 타임스탬프, 유효성 상태 등 **환경 메타데이터**를 함께 반환해야 합니다. (예: *"경고: 리포트가 5분 전 생성되었으며 최근 파일 수정이 반영되지 않음"*). 이를 통해 에이전트는 불필요한 분석을 멈추고 `npm test`를 먼저 실행하는 등 올바른 컨텍스트를 유지할 수 있습니다.

## 3. '추론 근거(Rationale)'의 투명한 노출 (Transparency of Rationale)
*   **문제**: 변수를 함수로 오인하여 "주석 없음" 경고를 띄울 때, 결과만 전달받은 에이전트는 파서의 내부 오류를 즉각적으로 인지하기 어렵습니다.
*   **원칙**: 위반 사항을 보고할 때 단순히 결과 메시지만 주지 말고, 도구가 판단한 **타입 정보 및 근거**를 포함해야 합니다. (예: `심볼 타입: Function(인식됨) -> 한글 주석 필수`). 에이전트가 "이것은 변수인데 왜 함수로 인식했나?"라고 즉시 파악하여 도구 자체의 결함을 빠르게 교정할 수 있도록 도와야 합니다.

## 4. 실시간 파라미터 튜닝 인터페이스 제공 (Dynamic Parameter Tuning)
*   **문제**: 보안 임계값(Entropy)이나 파일 길이 제한 등을 조정하기 위해 에이전트가 도구의 소스 코드를 직접 열어 수정하고 리빌드하는 과정은 매우 비효율적입니다.
*   **원칙**: MCP 도구의 Input Schema에 `security_threshold`, `max_lines`와 같이 **동적으로 설정을 주입할 수 있는 파라미터**를 설계해야 합니다. 에이전트가 코드를 건드리지 않고도 도구의 엄격도를 조절하며 유연하게 가설 검증(Trial & Error)을 수행할 수 있어야 합니다.

---
> **요약**: MCP를 위한 도구는 단순한 '단방향 실행기'를 넘어, LLM에게 현재 환경의 컨텍스트(Freshness)와 판단 근거(Metadata)를 양방향 데이터로 제공하는 **'지능형 센서(Intelligent Sensor)'** 역할을 수행해야 합니다.
